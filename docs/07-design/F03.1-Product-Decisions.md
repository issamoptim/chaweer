# F-03.1 — Product Decisions

> **Status:** For Approval
> **Source:** F03-Frontend-Review.md (Sections 15 & 16)
> **Purpose:** Freeze all product decisions resulting from the F-03 Frontend Review before implementation begins. Each recommendation from the review is classified as Approved, Rejected, or Deferred. Only Approved decisions will be implemented in F-03.1.

---

## 1. Decision Framework

Each decision in this document follows this structure:

| Field | Description |
|-------|-------------|
| **ID** | Unique decision identifier (PD-XX) |
| **Decision** | The decision statement — what we are deciding, not how engineers will implement it |
| **Source** | Which review recommendation(s) this decision addresses |
| **Status** | Approved / Rejected / Deferred |
| **Rationale** | Why this status was chosen |
| **Functional Impact** | What changes from a product, user, or business perspective |
| **Dependencies** | What must exist or be decided first |

Technical guidance for engineers (component names, API schemas, service methods, file paths) is intentionally excluded from the decision tables. It is consolidated in **Section 7 — Implementation Notes Appendix** to keep the product decisions durable and architecture-agnostic.

### Status Definitions

- **Approved**: This decision is confirmed and will be implemented in F-03.1.
- **Rejected**: This recommendation is not adopted. It will not be implemented.
- **Deferred**: This decision is valid but will not be implemented in F-03.1. It is postponed to a future milestone (F-04, F-05, or later).

## 2. Critical Improvements (Review Section 15 — Critical)

### PD-01: Add consultation title and description fields

| Field | Value |
|-------|-------|
| **ID** | PD-01 |
| **Decision** | The consultation offer must include a title and a description. Both are mandatory for publication. Without them, the profile cannot be published. |
| **Source** | Review Critical #1 |
| **Status** | **Approved** |
| **Rationale** | A consultation offer without a title or description does not give citizens enough information to understand what they are booking. These fields are the minimum clarity signal. The data model already supports them — this is a product validation decision, not a schema decision. |
| **Functional Impact** | The lawyer must provide a consultation title and description before their profile can be published. These appear on the offer page, in the offer preview, and on the future public profile. A profile with an untitled or undescribed consultation cannot be published. |
| **Dependencies** | None. |

### PD-02: Remove consultation duration entirely

| Field | Value |
|-------|-------|
| **ID** | PD-02 |
| **Decision** | Consultation duration is removed from the MVP product. The lawyer no longer selects a duration (15/30/45/60 min) when configuring their consultation offer. Duration is not a product concept in the MVP. |
| **Source** | Review Critical #2 |
| **Status** | **Approved** |
| **Rationale** | The platform provides a single consultation service. Duration adds friction without adding value. The consultation length is determined by the lawyer and client, not pre-configured. Removing it simplifies the offer configuration and the user experience. |
| **Functional Impact** | The offer page no longer presents a duration selector. The offer preview no longer shows a duration. Duration is not required for publication. The underlying data field is deprecated (not deleted) to avoid unnecessary migrations — it may be repurposed in a future milestone if the product evolves. |
| **Dependencies** | None. |

### PD-03: Change modalities from VIDEO/OFFICE to VIDEO/AUDIO/CHAT

| Field | Value |
|-------|-------|
| **ID** | PD-03 |
| **Decision** | The consultation offer supports three modalities: Video, Audio, and Chat. The previous modalities (Videoconference, Cabinet/Office) are replaced. At least one modality must be selected. Multiple modalities are allowed. |
| **Source** | Review Critical #3 |
| **Status** | **Approved** |
| **Rationale** | The MVP focuses on remote consultations. Office (in-person) is removed. Audio and Chat are added as lower-barrier alternatives for citizens who may not want video. This broadens accessibility and increases the likelihood of booking. |
| **Functional Impact** | The lawyer selects one or more consultation modalities from Video, Audio, Chat. At least one is required for publication. The selected modalities appear on the offer preview and the future public profile. Office/Cabinet is no longer offered. |
| **Business Rule** | At least one consultation modality must be selected. Multiple modalities are allowed. The lawyer can offer Video, Audio, and/or Chat consultations. |
| **Dependencies** | None. |

## 3. Important Improvements (Review Section 15 — Important)

### PD-10: Add Publication Readiness indicator

| Field | Value |
|-------|-------|
| **ID** | PD-10 |
| **Decision** | Add a Publication Readiness indicator that shows the percentage of completed mandatory requirements and a checklist of missing publication requirements. The indicator serves as a guidance tool, not just a score. |
| **Source** | Review Important #10 |
| **Status** | **Approved** |
| **Rationale** | Publication Readiness is the central product concept of the onboarding. Without a visual indicator, lawyers cannot gauge how close they are to publishing. | **Functional Impact** | The lawyer sees a Publication Readiness panel that displays: (1) a percentage (0–100%), (2) a checklist of all publication requirements with completed/missing status (✓/✗), (3) the list of remaining requirements below the percentage. Example: "Publication Readiness 82% — Remaining requirements: ✓ First name, ✓ Last name, ✓ Biography, ✗ Consultation title, ✗ Consultation description, ✗ Consultation price, ✗ Consultation modality". The indicator replaces or augments the 3-step stepper. The objective is to guide the lawyer toward publication instead of simply displaying a score. |

## 5. Additional Product Decisions (Beyond Review Recommendations)

These decisions formalize concepts that are implicit in the review but not listed as numbered recommendations.

### PD-29: Profile status field (DRAFT / PUBLISHED)

| Field | Value |
|-------|-------|
| **ID** | PD-29 |
| **Decision** | The professional profile has a status lifecycle: DRAFT (default) and PUBLISHED. Publication transitions DRAFT → PUBLISHED. The MVP uses only these two statuses. Additional statuses exist in the data model for future phases but are not used. |
| **Source** | Review Sections 4 (Business Rules), 7 (Publication Readiness Assessment) |
| **Status** | **Approved** |
| **Rationale** | The publication concept requires a profile status to determine marketplace visibility, sidebar items, and UI state. Without it, the publish button has no effect. The data model already supports this — no schema change is needed. |
| **Functional Impact** | A new profile starts as DRAFT. It is invisible on the marketplace. When the lawyer clicks "Publier mon profil" and all requirements are met, the status transitions to PUBLISHED. The profile becomes visible on the marketplace. The sidebar and UI adapt to the new status. |
| **Dependencies** | None. |
| **ARCHIVED status** | An `ARCHIVED` status is **intentionally deferred**. It would allow professionals to temporarily or permanently retire their profile while preserving data. However, introducing it now would add complexity to the visibility rules, marketplace queries, and UI state management without serving the publication KPI. `ARCHIVED` will be considered when the marketplace is mature enough to need profile lifecycle management (post-F-08). The existing `UNPUBLISHED` status can serve as an intermediate state if needed. |

### PD-30: Publication action endpoint

| Field | Value |
|-------|-------|
| **ID** | PD-30 |
| **Decision** | Publishing a profile is a dedicated action with server-side validation. The system validates all mandatory fields before transitioning the profile to PUBLISHED. If validation fails, the system returns a structured list of missing requirements so the lawyer can be guided to complete them. |
| **Source** | Review Section 4 (Business Rules P-02, P-03) |
| **Status** | **Approved** |
| **Rationale** | Publication is a state transition with server-side validation. It cannot be a simple field update. The system must enforce publication requirements even if the client-side validation is bypassed. The structured error response allows the UI to display exactly which requirements are missing, aligning with Principle P6 (guide, don't block). |
| **Functional Impact** | When the lawyer clicks "Publier mon profil", the system checks all 9 mandatory requirements server-side. If all pass, the profile is published (status → PUBLISHED, publication timestamp recorded). If any fail, the lawyer sees a list of the specific missing requirements (e.g., "Biography, Consultation title, Consultation modality") and is guided to complete them. The profile is not published until all requirements are met. |
| **Dependencies** | PD-29 (profile status). |

### PD-31: Publication Readiness calculation

| Field | Value |
|-------|-------|
| **ID** | PD-31 |
| **Decision** | Publication Readiness is a UX indicator calculated from the publication requirements. It does not require a dedicated API endpoint. The exact weighting algorithm is not hardcoded in this document — it can evolve as the product matures without changing the product philosophy. The initial implementation uses a simple binary check per requirement (completed = yes/no) with equal visual weight. |

The weighting strategy may evolve (e.g., weighted by trust impact, grouped by category) but the product philosophy remains: show the lawyer what's needed to publish. |
| **Functional Impact** | The Publication Readiness indicator evaluates 9 publication requirements: first name, last name, bar association, biography (>= 200 chars), at least 1 Legal Practice Domain, consultation title, consultation description, consultation price, at least 1 consultation modality. For each requirement, the indicator shows whether it is completed or missing. The percentage and checklist update in real-time as the lawyer fills in the form. |
