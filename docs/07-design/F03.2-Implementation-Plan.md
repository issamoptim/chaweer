## Epic 1 — Consultation Offer

**Decisions**: PD-01 (Add title + description), PD-02 (Deprecate duration), PD-03 (Change modalities)

### Files Affected

**Frontend**
- `apps/web/src/features/professional/pages/ProfessionalOfferPage.tsx` — Add title/description inputs, remove `SegmentedControl` duration, replace modality options (VIDEO/AUDIO/CHAT), update `canFinish` logic, update mutation payload
- `apps/web/src/features/professional/components/OfferPreviewCard.tsx` — Add title/description display, remove duration display, replace OFFICE icon with AUDIO/CHAT icons
- `apps/web/src/features/professional/components/SegmentedControl.tsx` — No longer used by offer page (can be kept for future use or removed)
- `apps/web/src/features/professional/types/professional-types.ts` — Update `ConsultationModality` type from `"VIDEO" | "OFFICE"` to `"VIDEO" | "AUDIO" | "CHAT"`, add `title` and `description` to `UpdateOfferInput`, remove `durationMinutes` from `UpdateOfferInput`
- `apps/web/src/features/professional/hooks/useUpdateOffer.ts` — No logic change, but `UpdateOfferInput` type changes propagate
- `apps/web/src/features/professional/services/professional-service.ts` — `updateOffer` method payload changes (title, description, modalities; no duration)

**Backend**
- `apps/api/prisma/schema.prisma` — Update `ConsultationModality` enum: remove `OFFICE`, add `AUDIO`, `CHAT` (requires Prisma migration)
- `apps/api/src/modules/professional/professional.schema.ts` — Rewrite `updateOfferSchema`: add `title` (required, non-empty string), `description` (required, non-empty string, max 500), remove `durationMinutes`, change `modalities` enum to `['VIDEO', 'AUDIO', 'CHAT']`
- `apps/api/src/modules/professional/professional.service.ts` — Update `setOffer`: persist `title` and `description` (update + create), stop persisting `durationMinutes`, remove hardcoded `title: 'Consultation'` on create
- `apps/api/src/modules/professional/professional.types.ts` — `ConsultationOfferData.durationMinutes` can remain (deprecated) but `ConsultationModality` import type changes

**Shared**
- `packages/shared/src/enums/consultation-modality.enum.ts` — Add `AUDIO: "AUDIO"` and `CHAT: "CHAT"`, remove `OFFICE: "OFFICE"`
- `packages/shared/src/contracts/professional.contract.ts` — `ConsultationOfferResponse.modalities` type already uses `ConsultationModality[]` (auto-updates from enum change)

**API**
- `PUT /professional/offer` — Request body changes: add `title`, `description`; remove `durationMinutes`; `modalities` accepts `VIDEO | AUDIO | CHAT`

**Database**
- Prisma migration: `ConsultationModality` enum change (remove `OFFICE`, add `AUDIO`, `CHAT`). Existing data with `OFFICE` must be migrated or handled.
- `ConsultationOffer.title` and `ConsultationOffer.description` already exist in schema (nullable) — no migration needed for PD-01.

**Validation**
- `updateOfferSchema`: `title` = `z.string().trim().min(1).max(120)` (required), `description` = `z.string().trim().min(1).max(500)` (required), `modalities` = `z.array(z.enum(['VIDEO', 'AUDIO', 'CHAT'])).min(1)`, `durationMinutes` removed
- Shared schema tests must be updated to reflect new offer schema

### Risks

- **Enum migration**: Removing `OFFICE` from the Prisma enum requires a database migration. If any existing rows have `OFFICE` in the `modalities` array, PostgreSQL will reject the migration. Must either (a) migrate existing data first, or (b) run a data cleanup script before the migration. Since this is pre-launch, the database likely has no production data — but dev/staging databases may need cleanup.
- **Type propagation**: The `ConsultationModality` type is used in multiple places (shared enum, frontend types, backend types, Prisma generated client). All must be regenerated and verified.
- **OfferPreviewCard**: Currently imports `MapPin` for OFFICE icon. Must add icons for AUDIO (e.g., `Phone`) and CHAT (e.g., `MessageCircle`).

### Acceptance Criteria

- [ ] Offer page has title and description input fields (mandatory)
- [ ] Offer page no longer shows duration selector
- [ ] Offer page shows three modality options: Video, Audio, Chat (no Office)
- [ ] At least one modality must be selected to proceed
- [ ] Offer preview card shows title, description, price, and selected modalities (no duration)
- [ ] `PUT /professional/offer` accepts and persists `title`, `description`, `modalities` (VIDEO/AUDIO/CHAT)
- [ ] `updateOfferSchema` validates new fields correctly
- [ ] `setOffer` service persists `title` and `description` on both create and update
- [ ] Shared `ConsultationModality` enum updated and propagated
- [ ] Prisma migration for enum change succeeds

### Complexity: **M**

### Dependencies

- None (can start immediately). However, the Prisma migration should be coordinated with Epic 4 (PD-33 seed script) if both touch the database in the same deployment.

- `apps/web/src/features/professional/api/professional-keys.ts` — May need cache invalidation on publish success

**Backend**
- **New file**: `apps/api/src/modules/professional/professional.publish.controller.ts` (or add to existing controller) — `publishProfileController`
- `apps/api/src/modules/professional/professional.service.ts` — Add `publishProfile(userId)` function: validate all 9 requirements, transition `status` to `PUBLISHED`, set `publishedAt`, return updated profile. On failure, throw structured error with `missingRequirements` array.
- `apps/api/src/modules/professional/professional.routes.ts` — Add `POST /professional/profile/publish` route
- `apps/api/src/modules/professional/professional.schema.ts` — No request body schema needed (empty body, uses auth userId)
- `apps/api/src/modules/professional/professional.types.ts` — Add `PublicationErrorData` type with `missingRequirements` array
- `apps/api/src/core/errors/` — May need a custom error class or use existing `ValidationError` with structured fields

**Shared**
- `packages/shared/src/contracts/professional.contract.ts` — `PublishResponse` already defined (status + publishedAt). Add `PublicationRequirementsMissing` error contract.

**API**
- **New endpoint**: `POST /professional/profile/publish` — No request body. Validates all 9 mandatory requirements server-side. On success: returns `{ success: true, data: { status: "PUBLISHED", publishedAt: "..." } }` with HTTP 200. On failure: returns HTTP 422 with `{ success: false, error: { code: "PUBLICATION_REQUIREMENTS_MISSING", missingRequirements: ["biography", "consultation_title", ...] } }`.

**Database**
- No schema changes. `ProfessionalProfile.status` and `publishedAt` already exist.

**Validation**
- Server-side validation in `publishProfile` function (not Zod schema — direct field checks):
  1. `user.firstName` non-empty
  2. `user.lastName` non-empty
  3. `profile.barAssociationId` non-null
  4. `profile.bio` length >= 200
  5. At least 1 `ProfessionalSpecialization` row
  6. `offer.title` non-empty
  7. `offer.description` non-empty
  8. `offer.price` > 0
  9. `offer.modalities` array length >= 1

### Risks

- **Publication Readiness sync**: The frontend `usePublicationReadiness` hook computes readiness client-side. The server-side `publishProfile` is the authoritative check. If they diverge (e.g., frontend says 100% but server rejects), the UX must handle the 422 gracefully and display the server's `missingRequirements`.
- **Success screen extraction**: The success screen is currently inline in `ProfessionalOfferPage.tsx` (lines 83-117). Extracting it to a separate component changes the page's render flow. Must ensure the `finished` state is replaced by a `published` state that shows the new `SuccessScreen` component.
- **Error handling**: The 422 response structure must be parsed by the frontend and displayed as a user-friendly list. The `apiClient` may need to expose the error response body.
- **Profile status in frontend**: The frontend `ProfessionalProfileData.status` field is already typed as `string`. The UI must check for `"DRAFT"` vs `"PUBLISHED"` to conditionally render. This is fragile — consider typing it as `ProfessionalProfileStatus` from the shared enum.

### Acceptance Criteria

- [ ] `POST /professional/profile/publish` endpoint exists and validates all 9 requirements
- [ ] On success, profile status transitions to `PUBLISHED` and `publishedAt` is set
- [ ] On failure, endpoint returns HTTP 422 with `missingRequirements` array
- [ ] Frontend `usePublicationReadiness` hook computes percentage and checklist from profile data
- [ ] `PublicationReadinessIndicator` component displays percentage and checklist
- [ ] CTA changes from "Terminer la configuration" to "Publier mon profil" when readiness = 100%
- [ ] Clicking "Publier mon profil" calls the publish endpoint
- [ ] On 422 response, frontend displays the list of missing requirements
- [ ] On success, the new `SuccessScreen` component is shown with publication confirmation
- [ ] Success screen shows enrichment CTAs (photo, domains, situations, languages, office)
- [ ] Success screen does NOT mention verification
- [ ] Success screen does NOT show disabled "Bientot" button

### Complexity: **L**

### Dependencies

- **PD-01, PD-02, PD-03** (Epic 1) must be completed first — the publication validation checks for `offer.title`, `offer.description`, and `offer.modalities` which are added by Epic 1.
- **PD-04, PD-05** (Epic 3) must be completed first — the publication validation checks for `barAssociationId` (PD-04) and `bio >= 200` (PD-05).
- **PD-06** (Epic 4) must be completed first — the publication validation checks for at least 1 specialization (PD-06).
- **PD-33** (Epic 4) must be completed first — the taxonomy must be seeded for expertise to work.

- `apps/web/src/features/professional/types/professional-types.ts` — No type changes needed (fields already exist)

**Backend**
- `apps/api/src/modules/professional/professional.service.ts` — No changes for PD-04/PD-05 (validation is enforced on publication endpoint in Epic 2, not on profile update). The `updateProfile` function already accepts `barAssociationId` and `bio`.
- `packages/shared/src/schemas/professional.schema.ts` — `updateProfileSchema` already accepts `bio` and `barAssociationId`. No changes needed for the PATCH endpoint. (Publication validation is separate.)

**Shared**
- No changes needed. `updateProfileSchema` already handles these fields.

**API**
- No API changes for this epic. The `PATCH /professional/profile` endpoint already accepts all fields. Mandatory enforcement happens on publication (Epic 2).

**Database**
- No changes. `barAssociationId` and `bio` already exist on `ProfessionalProfile`.

**Validation**
- Frontend-only inline validation:
  - Bar association: required (non-empty) — show error on blur if empty
  - Bio: minimum 200 characters — show character count "X/200 caractères minimum" and error on blur if < 200
- `canContinue` logic update: `form.firstName && form.lastName && form.barAssociationId && form.bio.length >= 200`

### Risks

- **ProTextarea/ProSelect API changes**: Adding `error`, `minLength`, `hint`, and `required` props to shared form components may affect other pages that use them. Must verify backward compatibility — all new props should be optional.
- **Bio validation UX**: The current `ProTextarea` shows `{value.length}/{maxLength}` (e.g., "0/600"). With a 200-char minimum, the display must show both the minimum threshold and the maximum. Example: "200 caractères minimum · 0/600".
- **canContinue change**: Currently `canContinue` only checks `firstName` and `lastName`. Adding bar association and bio checks may prevent navigation for existing profiles that don't have these fields yet. Must handle gracefully (pre-fill from profile data, show warnings not blocks if data is missing from an existing profile).

### Acceptance Criteria

- [ ] Bar association select shows red asterisk and "Obligatoire pour la publication" hint
- [ ] Bio textarea shows "200 caractères minimum" indicator and error on blur if < 200
- [ ] Bio placeholder text updated to "Presentez votre approche, vos succes notables, ce qui vous distingue"
- [ ] Contextual trust hints appear below bar association and bio fields
- [ ] Profile page shows mandatory fields (name, bar association, bio) before optional fields (photo, professional title)
- [ ] `canContinue` requires bar association and bio >= 200 chars
- [ ] `ProTextarea` accepts optional `error`, `minLength` props
- [ ] `ProSelect` accepts optional `required`, `error` props
- [ ] `ProInput` accepts optional `error`, `hint` props

### Complexity: **M**

### Dependencies

- None (can start immediately). However, the `canContinue` change affects the flow into Epic 2 (publication). If bar association and bio are not filled, the lawyer cannot proceed to expertise — which is the intended behavior.
