# US-007.1 — Current User Endpoint

## Business Value

Permettre au frontend de récupérer les informations de l'utilisateur actuellement authentifié afin d'afficher son profil, adapter l'interface selon son rôle et maintenir un état cohérent côté client.

Cette User Story constitue le point d'entrée lecture de l'utilisateur connecté.

Elle n'introduit aucune logique métier supplémentaire.

Elle n'ajoute aucun nouveau modèle de données.

Elle ne modifie aucun comportement existant.

---

# Contexte

Le domaine Identity est implémenté (US-005).

L'authentification locale est opérationnelle (US-006).

L'authentification Google est implémentée (US-007).

Le middleware d'authentification JWT existe déjà.

Le modèle User existe avec ses champs actuels.

Cette User Story ajoute uniquement un endpoint de lecture.

---

# Objectif

Ajouter un endpoint REST permettant au frontend de récupérer les informations de l'utilisateur actuellement authentifié.

Cet endpoint est en lecture seule.

Aucun side-effect.

Aucune modification de données.

Aucune création de token.

Aucune interaction avec le Refresh Token.

---

# Contraintes

Ne modifier aucun comportement existant.

Ne pas toucher au login.

Ne pas toucher au refresh token.

Ne pas modifier le JWT.

Ne pas modifier le middleware d'authentification.

Ne pas modifier le modèle User.

Ne pas modifier Prisma.

Ne pas modifier Docker.

Aucun changement fonctionnel.

---

# Endpoint

GET /auth/me

---

# Authentification

Bearer Access Token

Header :

Authorization: Bearer <access_token>

L'Access Token est le JWT généré lors du login ou du refresh.

Aucun autre mécanisme d'authentification n'est accepté.

---

# Réponse

## 200 OK

```json
{
  "success": true,
  "data": {
    "id": "...",
    "role": "CLIENT",
    "email": "user@example.com",
    "firstName": "...",
    "lastName": "...",
    "avatarUrl": null
  }
}
```

### Champs retournés

id

role

email

firstName

lastName

avatarUrl

### Champs non retournés

passwordHash

authProvider

status

createdAt

updatedAt

Aucun autre champ ne doit être exposé.

---

## 401 Unauthorized

```json
{
  "success": false,
  "error": {
    "code": "UNAUTHORIZED",
    "message": "Authentification requise."
  }
}
```

---

# Codes HTTP

200 OK

401 Unauthorized

Aucun autre code HTTP n'est attendu pour ce endpoint.

---

# Codes d'erreur

UNAUTHORIZED

INTERNAL_ERROR

Aucun autre code d'erreur n'est utilisé par ce endpoint.

---

# Règles métier

## Lecture seule

Le endpoint ne modifie aucune donnée.

Aucun side-effect.

Aucune écriture en base.

Aucune journalisation métier.

---

## Authentification obligatoire

Le endpoint exige un Access Token valide.

Sans token : 401.

Token expiré : 401.

Token invalide : 401.

Aucun accès anonyme.

---

## Utilisateur courant

Le endpoint retourne uniquement les informations de l'utilisateur identifié par le JWT.

Aucun paramètre de requête.

Aucun identifiant dans l'URL.

L'utilisateur est déterminé exclusivement à partir du payload du JWT.

---

## Format de réponse

La réponse respecte le format standard du projet.

Succès :

```json
{
  "success": true,
  "data": {}
}
```

Erreur :

```json
{
  "success": false,
  "error": {
    "code": "...",
    "message": "..."
  }
}
```

---

## Sécurité

Le mot de passe ne doit jamais être retourné.

Le hash du mot de passe ne doit jamais être retourné.

Aucune information sensible ne doit être exposée.

Seuls les champs définis dans le contrat API sont retournés.

---

## Réutilisation de l'existant

Le endpoint réutilise :

- Le middleware d'authentification JWT existant
- Le Prisma Client existant
- Le gestionnaire d'erreurs centralisé
- Le format de réponse standard

Aucune duplication de logique n'est autorisée.

Aucun nouveau service de sécurité n'est créé.

# Tests

## Utilisateur authentifié valide

Étant donné un utilisateur authentifié avec un Access Token valide,

Lorsqu'il appelle GET /auth/me,

Alors la réponse est 200 OK,

Et la réponse contient uniquement les champs : id, role, email, firstName, lastName, avatarUrl.

---

## Token manquant

Étant donné une requête sans header Authorization,

Lorsqu'elle appelle GET /auth/me,

Alors la réponse est 401 Unauthorized,

Et le code d'erreur est UNAUTHORIZED.

---

## Token invalide

Étant donné une requête avec un token malformé,

Lorsqu'elle appelle GET /auth/me,

Alors la réponse est 401 Unauthorized,

Et le code d'erreur est UNAUTHORIZED.

---

## Token expiré

Étant donné une requête avec un token expiré,

Lorsqu'elle appelle GET /auth/me,

Alors la réponse est 401 Unauthorized,

Et le code d'erreur est UNAUTHORIZED.

---

## Utilisateur Google

Étant donné un utilisateur authentifié via Google avec un Access Token valide,

Lorsqu'il appelle GET /auth/me,

Alors la réponse est 200 OK,

Et la réponse contient les mêmes champs qu'un utilisateur LOCAL.

---

## Aucun side-effect

Étant donné un utilisateur authentifié,

Lorsqu'il appelle GET /auth/me plusieurs fois consécutivement,

Alors chaque réponse est identique,

Et aucune donnée n'est modifiée en base.
